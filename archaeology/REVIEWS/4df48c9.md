The C bug was `dbEvent.c::db_queue_event_log` losing its early-exit compaction check: when `npend > 0` and both the queued event and the new event were reference-type (`!dbfl_has_copy`), the new event should be dropped (the consumer will see the freshest record-field value at process time anyway). Without the check, rapid posting unboundedly accumulated duplicate ref-events, overflowing the queue and dropping events that mattered.

In `epics-base-rs` there is no `dbEvent` ringbuffer with `dbfl_type_ref` / `dbfl_type_val` (copy-vs-ref) variants. The subscription machinery in `server/record/record_instance.rs::notify_field_with_origin` (line 1431) builds a full owned snapshot per post (`make_monitor_snapshot(value)`), serializes it to a `MonitorEvent { snapshot, origin }`, and pushes via per-subscriber bounded `mpsc::channel(64)` (line 1485). Overflow is handled by a deliberate "drop oldest, keep newest" coalesce slot:

```
if sub.tx.try_send(event.clone()).is_err() {
    if let Ok(mut slot) = sub.coalesced.lock() {
        *slot = Some(event);  // overwrite with newest
    }
}
```

(record_instance.rs:1414, 1447). Consumer drains via `pop_coalesced` after every successful `recv()` (`db_access.rs:286` `next_event`). This is the Rust-shaped equivalent of the C compaction: under burst, the consumer always lands on the freshest snapshot, the queue itself never grows beyond 64, and there are no live-pointer "ref" events whose semantics requires deduplication. The C invariant ("don't queue dup ref events") is replaced by a stronger one ("the coalesce slot always holds the newest unread event"). Structurally absent.
