The C bug was in `modules/libcom/src/osi/os/posix/osdThread.c::epicsThreadGetCPUs`, which on POSIX called `sysconf(_SC_NPROCESSORS_ONLN)` and ignored the calling process's CPU affinity mask. The audit target `src/server/database/callback.rs::get_cpu_count` does not exist in `epics-base-rs` — there is no `callback.rs` module and no equivalent of `callbackParallelThreads`. The only CPU-count call site in the crate is `src/server/iocsh/commands.rs:467` (`iocStats`-style report) which already uses `std::thread::available_parallelism()`. Per Rust 1.74+, `available_parallelism()` consults `sched_getaffinity` on Linux, so the affinity-aware behavior the C fix added is already provided by the standard library. Tokio's worker count, when used, is also derived from this affinity-aware count by default. No `num_cpus::get_physical()` or raw `sysconf` call exists.
