The C bug is an init-ordering gap: `dbReadCOM` (called from `dbReadDatabase`/`dbReadDatabaseFP`) parses `.db` files, but a parse error invokes `errlogPrintf` which looks up symbols in `errSymTable`. `errlogInit` was not guaranteed to have been called first, so an early `iocsh`-driven `dbReadDatabase` could trip an uninitialized `errSymTable` lookup. The fix calls `errlogInit(0)` at the top of `dbReadCOM` (idempotent).

`crates/epics-base-rs` has no `static_lib.rs`, no `errSymTable`, no lazily initialized error-symbol table at all. Logging goes through `tracing` / `runtime/log.rs`, which is always safe to call regardless of init state. The DB loader entry points (`server/db_loader/mod.rs`) report parse errors via standard `Result<_, ParseError>` returns and `tracing::error!`, not via an EPICS error-symbol lookup. The init-ordering precondition the C fix establishes therefore has no Rust analogue to enforce.
