The C bug is `popFirstTemp()` in `dbLexRoutines.c` dereferencing `ellFirst(&tempList)` without a NULL check during yacc/bison-driven .dbd parsing recovery. base-rs has no equivalent: `crates/epics-base-rs/src/server/db_loader/mod.rs::parse_db` is a hand-rolled char-walking parser with explicit `expect_char`/`read_quoted_string` helpers that return `CaResult<...>` on every failure, and there is no temp-list LIFO stack used by the parser. A grep across the crate for `temp_list|TempList|tempList|popFirst|pop_first` returns no hits. The Rust parser never reaches a state where a NULL deref equivalent (i.e., `unwrap()` on an empty `Vec`/`VecDeque`) is possible because each lexical step either consumes input and advances or returns `Err(CaError::DbParseError)`. No fix needed.
