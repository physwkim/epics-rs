The C bug is in `compressRecord.c::compress_array`: the early-return guard `if (no_elements < n) return 1;` rejected sub-N input arrays unconditionally even when `PBUF=YES`. The fix gates the guard on `prec->pbuf != menuYesNoYES` and switches the averaging count to `no_elements`.

In `crates/epics-base-rs/src/server/records/compress.rs`, there is no `compress_array` function â€” `CompressRecord` only supports scalar pushes via `push_value(input: f64)` driven by `put_field("VAL", EpicsValue::Double(v))`. There is also no `pbuf` field, no waveform/array INP fan-in for the compress record, and no early-return guard that could reject sub-N input. The `EpicsValue::DoubleArray` branch in `put_field` simply replaces `self.val`, which is the output buffer, not an input window to compress. The defect is thus structurally absent.
