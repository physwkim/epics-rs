The C bug introduced `S_db_emptyArray` (M_dbAccess|69) so callers of `dbGet`/`dbDbGetValue` could distinguish "field type mismatch" from "zero-element array read" — the previous shared `S_db_badField` masked legitimate empty-array cases with spurious errors. In `epics-base-rs`, the equivalent surface is `EpicsValue::*Array(vec![])` plus `Option<EpicsValue>` return types: empty arrays are a fully representable success value, and the discrimination C needed an error code for is structural. Concretely:
- `links.rs::read_link_value` returns `Option<EpicsValue>`; `None` is "no value", `Some(EpicsValue::DoubleArray(vec![]))` is "empty array" — already distinguishable.
- `db_access.rs::DbChannel::get_*` returns scalar fallbacks via `unwrap_or(0.0)` after `to_f64()`; arrays don't flow through these scalar accessors.
- `field_io.rs::put_pv` (lines 79–86) explicitly rejects empty arrays into scalar fields with `CaError::InvalidValue` rather than silent coercion (the 12cfd41 audit fix).

There is no `S_db_emptyArray` status code to add and no shared `BadField` enum variant collapsing the two cases. Structurally absent.
