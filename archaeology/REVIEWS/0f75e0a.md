The C bug was that `dbDbLink.c::processTarget` used `assert()` to verify the `procThread` invariant before/after `dbProcess(pdst)`; on a logic error the IOC would abort instead of logging. There was also a copy-paste error checking `psrc->procThread` twice. The audit target `src/server/database/dbDbLink.rs::process_target` does not exist in `epics-base-rs` — there is no `dbDbLink.rs` module. The DB-link processing path is implemented directly in `src/server/database/processing.rs` (`process_record_with_links_inner` and the CP-target propagation block at lines 1321-1357). It uses no `procThread` assert: recursion is gated by an async-aware `processing: AtomicBool` on `RecordInstance` (paired with `RPRO`/`PUTF` flags) plus a `visited: HashSet<String>` for cycle detection. Logic errors degrade to early returns, not panics — there is no `assert!` / `unwrap()` on the recursion-guard path. The C-shaped abort bug cannot occur.
