# Review: 235f8ed

## Verdict
not-applicable — structurally absent in base-rs.

## Analysis
The C bug is that `struct db_field_log` lacked an originating-event
mask byte, so filters checking `DBE_PROPERTY` always saw `mask == 0`
and could not distinguish property events from value events. The fix
added `unsigned char mask` and set it from `pevent->select` in
`db_create_event_log`.

`epics-base-rs` does not have an analogous `db_field_log` struct. The
posting-mask is already plumbed end-to-end on `ProcessSnapshot`:

- `ProcessSnapshot.event_mask: EventMask`
  (`server/record/record_trait.rs:152`).
- `EventMask` carries `VALUE/LOG/ALARM/PROPERTY` bits
  (`server/recgbl.rs:30-56`).
- `notify_from_snapshot`
  (`server/record/record_instance.rs:1397-1423`) intersects
  `posting_mask` with each subscriber's mask before delivering, and
  `notify_field_with_origin` (line 1431-1456) does the same for
  ad-hoc field notifications.

Any subscriber filter (downstream CA/PVA layer) sees the correct
posting-mask via the event delivery path, so the structural cause of
the C bug — losing the mask between the source and the filter —
cannot exist here.
