The C bug was `dbDeleteRecord()` failing to free the heap allocations held by each `DBLINK` struct (link strings, JSON bodies) before freeing the record node, leaking on every dynamic record removal. base-rs has no record-deletion API: `PvDatabase` (`crates/epics-base-rs/src/server/database/mod.rs`) exposes record creation/lookup but no `delete_record` / `remove_record` path; the `records` map is never selectively pruned at runtime. All link-related state lives in owned Rust types (`ParsedLink`, `String`, `EpicsValue`) inside the `RecordInstance`, so when records are eventually dropped (only at process exit / `Drop` of `Arc<RwLock<RecordInstance>>`) the link-internal allocations are released by the standard `Drop` chain — no manual free helper required. If a future commit adds a runtime `delete_record` path, the audit reduces to "ensure the `Arc` is fully released so `RecordInstance::Drop` runs", which Rust's borrow checker enforces automatically. Closing as not-applicable.
