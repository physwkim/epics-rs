The C fix replaced `errlog`'s single ring buffer + lock-during-print design with a double-buffer scheme (`bufs[2]`) so producers never serialize on the consumer's I/O. base-rs has no `errlog` subsystem: `crates/epics-base-rs/src/runtime/log.rs` defines four trivial `rt_debug!`/`rt_info!`/`rt_warn!`/`rt_error!` macros that go directly to `eprintln!`. There is no shared queue, no listener registry, no flush counter, no consumer thread holding any lock — each producer's call site does its own write. Producers cannot serialize on a consumer because there is no consumer thread. The lock-contention bug pattern is structurally absent. If base-rs later adopts a centralized buffered logger (e.g. ring-buffered `errlog`-style sink for IOC console + remote listeners) the right pattern is the one summarized in the analysis: `tokio::sync::mpsc` (or `std::sync::mpsc`) for decoupled producer/consumer, atomic generation counter for `errlog_flush()` synchronization, and never holding the queue lock across stderr writes. Closing as not-applicable for current code.
