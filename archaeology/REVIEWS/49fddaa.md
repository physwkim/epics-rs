The C bug was in `errlog.c::errlogRemoveListeners` / `errlogThread`: a listener callback that called `errlogRemoveListeners` to deregister itself caused `ellDelete` to free the linked-list node currently being iterated under `listenerLock`, and the worker then read `ellNext()` from a freed pointer (heap corruption / segfault, typically during IOC shutdown). The fix added `active` / `removed` bit-fields to `listenerNode` and a deferred-free protocol. The audit target `crates/epics-base-rs/src/log/errlog.rs` does not exist in base-rs. There is no errlog listener registry: `crates/epics-base-rs/src/runtime/log.rs` is just four `eprintln!` macros with no listener list, no worker thread, no `errlogAddListener` / `errlogRemoveListeners`, and no callback-dispatch loop that could be re-entered. A workspace grep for `errlog`, `listenerNode`, `ellDelete`, `errlogAddListener` finds zero hits in src. The Rust ownership model would also force the listener container to be reasoned about explicitly (e.g. `Vec<…>::retain` after the dispatch loop, not during) the moment such a registry is added — but there is currently no surface area to mishandle. Closing as not-applicable; if a future commit adds an `errlog`-style listener registry with re-entrant callbacks, the deferred-removal pattern from the audit recommendation should be applied at that time.
