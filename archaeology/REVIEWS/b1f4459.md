The C bug was that DB link internals stored a flat `DBADDR*` instead of a `dbChannel*`, so `dbDbGetDBFtype` and `dbDbGetElements` returned the unfiltered field type and element count, ignoring channel filter plugins (e.g., array-slice) that mutate the effective type/count. The audit targets `db_db_get_elements` and `db_db_get_dbf_type` do not exist in `epics-base-rs/src/server/database/links.rs`. The Rust link representation is `ParsedLink` (Db/Ca/Pva/Constant/None) which carries only a record name, field name, and process policy — there is no DBADDR analog cached on the link. Reads route through `read_link_value` (`links.rs:19`) which calls `get_pv` / `resolve_external_pv` and returns the live, current `EpicsValue` from the target record. There is also no dbChannel filter plugin system in this crate (no array-slice / dbnd / sync filters), so the underlying source of the C bug — a layer between raw field and the channel-visible value — does not exist. Element count comes from the live `EpicsValue` length, which is always post-coercion and reflects what would be a "final" value in C.
