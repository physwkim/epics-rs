# Review: 059d32a — dbChannelOpen probe init

## Verdict
not-applicable — the C bug is `dbChannel.c::dbChannelOpen` constructing
a `db_field_log probe` by per-field assignment, leaving padding and
unlisted members indeterminate; filter plugins'
`channel_register_pre/post` callbacks then read garbage.

base-rs has no `dbChannel` / filter-chain machinery and no
`db_field_log` type. The high-level channel abstraction in
`src/server/database/db_access.rs` (`DbChannel`, `DbSubscription`)
operates over typed `EpicsValue` snapshots fetched through the
async PV interface — there is no probe-based filter plugin
registration that would inspect a partially-initialized struct, and
no `channel_register_pre/post` callback surface. There are no
`channel.rs` filter plugins in the crate (only the
`std::ops::BitOr*` `EventMask` flag in `recgbl.rs`). Any future port
of dbChannel would naturally use `Default::default()` or
`{ .. Default::default() }` which the analysis already calls out as
the safe Rust idiom.

## C reference
`modules/database/src/ioc/db/dbChannel.c:dbChannelOpen` — replaced
field-by-field init with `memset(&probe, 0, sizeof(probe))` plus
non-zero overrides.

## Build
No code changes; `cargo check -p epics-base-rs` already clean.
