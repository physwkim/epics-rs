The C bug was the loss of `prec->pact = TRUE` in `waveformRecord.c::process()`, allowing concurrent re-entry of the same record while async device support was still pending. base-rs's waveform record (`crates/epics-base-rs/src/server/records/waveform.rs`) is a passive data store with no `process()` entry point — processing is centralized in `crates/epics-base-rs/src/server/database/processing.rs::process_record_with_links_inner` and `crates/epics-base-rs/src/server/record/record_instance.rs::process_local`. The PACT-equivalent (`RecordInstance::processing: AtomicBool`) is set BEFORE `record.process()` either by `process_local` (line 1138-1141: `swap(true, AcqRel)` with reentrant-detect via LCNT) or by the central pipeline holding the record's `RwLock` write guard for the entire process call. Async writes set `processing.store(true, Release)` before spawning the completion task (`processing.rs:617-619`), and `complete_async_record` clears it on completion. The Rust port structurally cannot lose the PACT flag because the centralized pipeline owns it uniformly across record types — no per-record set is ever required. No fix needed.
