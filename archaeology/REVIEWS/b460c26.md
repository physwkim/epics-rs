# Review: b460c26 — DBF_MENU out-of-range numeric fallback

## Verdict
not-applicable — the C bug is in `dbFastLinkConv.c::cvt_menu_st` and
`dbStaticLib.c::dbGetStringNum` returning `S_db_badChoice` (or NULL)
for an `epicsEnum16` value that exceeds `pdbMenu->nChoice` (e.g. SSCN
defaulting to 65535), causing CA gets to fail.

base-rs has no menu-table / `dbFastLinkConv` machinery. The DB-field
type set is enumerated in `src/types/dbr.rs` and EPICS values are
carried as `EpicsValue` in `src/types/value.rs` — there is no
`DBF_MENU` variant, no `pdbMenu` lookup, no choice-index → string
conversion table. Enum-style fields like `SSCN` are stored as
high-level Rust enums (e.g. `ScanType` in `src/server/record/scan.rs`)
with `Display`/`FromStr` rather than indexed lookups, so the
"index >= n_choice" out-of-range condition cannot arise. There is no
`db_get_string_num` equivalent that could return an error or NULL on
an out-of-range index.

## C reference
- `modules/database/src/ioc/db/dbFastLinkConv.c:cvt_menu_st` — fall
  back to `epicsSnprintf("%u")` instead of `S_db_badChoice`.
- `modules/database/src/ioc/dbStatic/dbStaticLib.c:dbGetStringNum` —
  format raw integer for out-of-range; NULL guard for `pfield`.

No code changes; `cargo check -p epics-base-rs` already clean.
