C ref: `modules/ca/src/tools/tool_lib.c:use_ca_timeout_env` (commit 1d056c6) added a function reading `EPICS_CLI_TIMEOUT` from the environment and made `caget`/`caput`/`camonitor`/`cainfo` consult it before parsing `-w`, so a site default could be set in the shell environment without rewriting every invocation. ca-rs has no `tool_lib.rs` — each tool is a self-contained binary. Each of the four ca-rs CLI tools previously hard-coded `1.0` as the clap `default_value`/`default_value_t`, ignoring `EPICS_CLI_TIMEOUT` entirely. Changed `timeout: f64` to `Option<f64>` and added a small `env_default_timeout()` helper per binary that reads `std::env::var("EPICS_CLI_TIMEOUT").ok().and_then(|s| s.parse().ok()).unwrap_or(1.0)` and is used as the fallback when `-w` is not supplied. CLI `-w` continues to override (priority preserved). Unparsable env values silently fall back to `1.0` — matches the C behaviour where `epicsScanDouble` failure leaves the previous value in place (which here is the hardcoded default, since there is no prior config layer). `cargo check -p epics-ca-rs --bins` clean.
