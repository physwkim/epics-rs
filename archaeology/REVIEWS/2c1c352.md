The C change adds `dbPutStringSuggest(DBENTRY*, const char*)` in
`modules/database/src/ioc/dbStatic/dbStaticLib.c`, which iterates DBF_MENU
or DBF_DEVICE choices and calls `epicsStrSimilarity()` to print a
`Did you mean "<best>"?` hint after a `S_db_badChoice` error from
`dbRecordField` in `dbLexRoutines.c`.

base-rs has no `dbStaticLib`/`dbLexRoutines` analog. The audit target
`src/server/database/static_lib.rs::put_string_suggest` does not exist.
DBD parsing in this codebase is split across `server/db_loader/` (which
loads `.db` substitution + record stanzas, not menu/device validation
against a DBD registry) and `EpicsValue::parse` in
`crates/epics-base-rs/src/types/value.rs:611`, which validates against a
fixed `resolve_menu_string` table — there is no menu-choice list to
iterate per field, and DBF_DEVICE is not modeled as a typed field with
a choice table. Without that registry there is nothing to compute string
similarity against.

The original analysis itself classifies this as "Pure UX, not a bug" and
the C author wrote it as a usability improvement layered on top of an
already-correct DBD parser. base-rs has not yet built that parser layer,
so there is no valid surface for a "did you mean" hint. Not applicable.
