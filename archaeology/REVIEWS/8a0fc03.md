The C bug is `dbPutFieldLink()` calling `dbChannelOpen(chan)` without capturing the return into `status`, so a failed channel-open silently returns `S_db_Ok` from `dbPutFieldLink`. base-rs has no `dbPutFieldLink` / `dbChannelOpen` equivalent: link strings are stored as raw record fields (e.g. `INP`, `OUT`, `DOL`) and parsed lazily by `parse_link_v2` at access time inside `read_link_value` / `write_db_link_value` (database/links.rs:19, :122). There is no link-installation step that opens a channel and returns a status code that could be discarded. Audit of fallible `?`-style dispatch in `db_access.rs`/`field_io.rs` shows put paths consistently propagate `CaResult` via `return Err(...)` or `?` — there are no `let _ = fallible_call()` discards on the put-link path. No fix needed.
