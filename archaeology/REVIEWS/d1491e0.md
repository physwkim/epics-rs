The C change in `modules/database/src/ioc/db/dbTest.c::dbpf` replaces a
custom whitespace tokenizer (which over-allocated `MAX_STRING_SIZE` per
element and could not represent empty arrays / spaces-in-strings) with
`dbPutConvertJSON()`, using `dbValueSize(dbrType)` for per-element
allocation.

base-rs `dbpf` is implemented in
`crates/epics-base-rs/src/server/iocsh/commands.rs:117` (`cmd_dbpf`). It
is **scalar-only** — it takes a single `value: String` argument and
delegates parsing to `EpicsValue::parse(dbf, value_str)` (`types/value.rs:611`),
which has no array branch at all (no `[...]` recognition, no whitespace
split, no `serde_json` import). Confirmed via `Grep` for
`split_whitespace|json|serde_json|from_str.*Array` in `value.rs` (zero
hits). There is no broken whitespace tokenizer to replace, and no
`dbValueSize`-style allocation pattern: `EpicsValue` enum variants
(`ShortArray(Vec<i16>)`, `FloatArray(Vec<f32>)`, etc.) inherently encode
element type and size, so the C-era element-size bug class (`MAX_STRING_SIZE`
per element regardless of dbrType) cannot occur in this code.

Adding JSON-array put support to `cmd_dbpf` would be a net-new feature,
not a fix. The archaeology entry itself flags `rust_verdict: partial,
severity: low` with audit recommendation phrased as "if base-rs exposes
a dbpf-equivalent ... it should". The current implementation simply
does not yet expose array puts — a future-feature note, not a bug.
Not applicable.
