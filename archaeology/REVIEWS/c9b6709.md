# Review

## Verdict
**not-applicable** — no logClient/TCP send path exists to add a probe to.

## Evidence
- `audit_targets.file = src/log/log_client.rs` does not exist.
- `crates/epics-base-rs/src/runtime/log.rs` is the entire logging surface and
  uses `eprintln!` only — no TCP socket, no `send()`, no flush function, no
  backlog. There is no place to add a 0-byte probe or to configure
  `socket2::TcpKeepalive`.
- A grep of the crate for `logClient`, `LogClient`, `log_client` and TCP-related
  log shipping primitives returns no Rust implementation hits (only one comment
  in `record_instance.rs:747`).

## Decision
The half-open / silent-broken-pipe latency described in `logClient.c:logClientFlush`
cannot occur because no remote log-shipping over TCP is implemented.
If a `tokio::net::TcpStream`-backed log client is added later, configure
TCP keepalive via `socket2` before handing the socket to tokio, and check
write errors for `BrokenPipe`/`ConnectionReset` to trigger reconnect.
