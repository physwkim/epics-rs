The C bug was in `compressRecord.c::compress_array` where N-to-M compression with PBUF=YES (partial-buffer) advanced `psource` incorrectly past the available input — `nnew = no_elements / n` rounded the input down but the loop then re-iterated `nnew` times, and the median variant used `psource += nnew` instead of `psource += n`. base-rs `crates/epics-base-rs/src/server/records/compress.rs::CompressRecord` is a scalar-input rewrite: `push_value(input: f64)` accumulates one value at a time into `accum: Vec<f64>` and only emits a compressed value once `accum.len() >= self.n`. There is no array-input batch path (the `put_field("VAL", DoubleArray)` arm just replaces `self.val` wholesale; it does not run the N-to-1 algorithm over the array), no PBUF field, and no median variant. Only ALG codes 0/1/2/3 (Low/High/Mean/CircularBuffer) are implemented — no median bug surface. The `accum.clear()` after each emission means there is no possibility of a "partial group" being mis-indexed; partial accumulators simply stay buffered until the next push completes them, exactly the `PBUF=NO` discard-incomplete behaviour. Rust slice/`Vec` access is bounds-checked, so even a future array path could not read past available input without a panic. Closing as not-applicable; if a future change adds an array-batched compress path with PBUF semantics, the audit recommendation in the analysis MD should be revisited.
