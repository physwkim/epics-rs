The C bug was an unbounded `convert[dbrType]` array index: a malformed `dbrType` could read a function pointer past the end of the table. In `epics-base-rs`, constant-link parsing lives in `src/server/record/link.rs::parse_link_v2` and `ParsedLink::constant_value`. There is no integer-indexed function-pointer dispatch table; the link parser detects numeric vs. quoted-string constants by `f64::parse` / `starts_with('"')` and returns a typed `EpicsValue::Double(v)` or `EpicsValue::String(s)`. All field-type dispatch in `value.rs::convert_to` is via the `DbFieldType` enum (a Rust enum, not a `short`), which is exhaustively pattern-matched. There is no integer→pointer indexing path that could go out of bounds, so the OOB read / corrupted function-pointer call is structurally impossible.
