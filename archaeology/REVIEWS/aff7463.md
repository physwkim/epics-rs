The C bug was that `aaiRecord.c::process` and `aaoRecord.c::process` did not call `db_post_events` for `NORD` after device support changed the element count, so monitors on `aai.NORD`/`aao.NORD` went stale. In `epics-base-rs`, neither the `aai` nor the `aao` record type is implemented — `src/server/records/` contains `ai`, `ao`, `bi`, `bo`, `waveform`, `mbbi`, `mbbo`, etc., but no `aai.rs` or `aao.rs`. The audit targets `src/server/database/rec/aai.rs` and `rec/aao.rs` do not exist (and the parent `database/rec/` directory does not exist). The companion array record `waveform` in base-rs has no `process()` method at all (it is a passive read-from-VAL container) and updates `NORD` directly inside `put_field` (`src/server/records/waveform.rs:226-258`), so the missing-post-from-process pattern does not arise. The bug class is structurally absent.
