The C bug is `dbGet` using `!pfl` instead of `!dbfl_has_copy(pfl)` to choose between the live record pointer and the field-log copy, which races on `dbfl_type_ref`+`dtor==NULL` borrowed pointers without the scan lock. base-rs has no `db_field_log` construct: subscriptions deliver `MonitorEvent { snapshot: Snapshot, ... }` (server/pv.rs:131) where `Snapshot` (server/snapshot.rs:51) holds an owned `EpicsValue` (always-owned variant: `Double(f64)`, `String(String)`, owned `Vec<...>` arrays, etc.). The producer side (`notify_field_*` in record_instance.rs) clones the value while holding the record lock, so the value handed to subscribers is a fully-owned copy by construction. No "borrowed-from-record" alternative exists, so the predicate-mismatch class of bug cannot occur. No fix needed.
