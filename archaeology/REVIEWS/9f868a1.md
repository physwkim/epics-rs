The audit target `src/server/database/db_event.rs::db_cancel_event` does not exist in `epics-base-rs`. There is no `db_event.rs`, no `pflush_sem`/`pflush_seq` global semaphore, no `evUser->waiters` ELLLIST, and no event-task flush-cycle synchronization. Subscription cancellation in the rewrite is per-subscriber RAII via `DbSubscription::drop` (`src/server/database/db_access.rs:329-356`), which spawns an independent `remove_subscriber(sid)` task per drop. Concurrent cancels are inherently independent — each Drop has its own future, no shared semaphore mediates them — so the "thundering herd on shared `pflush_sem`" deadlock the C fix targets cannot occur. No per-waiter notify infrastructure is required because there is no shared signal point to begin with.
