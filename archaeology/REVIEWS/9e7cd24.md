The C bug was that `mbbiRecord.c::special` and `mbboRecord.c::special` posted `DBE_PROPERTY` themselves on enum-string field writes, gated on `val == fieldIndex - ZRST` so the property event was lost when the changed string wasn't the active one. The fix removes that ad-hoc post and relies on the central `dbAccess.c::dbPut` property-event path (paired with b7cc33c).

In `epics-base-rs`, `crates/epics-base-rs/src/server/records/mbbi.rs` and `mbbo.rs` do not implement `Record::special` (the trait default in `record_trait.rs:278` is a no-op). Neither file calls `notify_field` / `notify_field_with_origin` for `ZRST`–`FFST` writes, and there is no per-record `db_post_events(DBE_PROPERTY)` call anywhere in the mbbi/mbbo logic. Property-class metadata (enum strings) is surfaced to monitors through `RecordInstance::populate_enum_info` + the metadata cache, invalidated by `notify_field_written` at the central `field_io.rs` write path — i.e. the architecture the C fix points toward.

The defect (record-local `special` racing the central path) is structurally absent in the Rust rewrite.
