The C fix made `epicsInt8` `signed char` everywhere so DBF_CHAR sign-extends
correctly when widened. epics-rs picked up the analogous fixes incrementally:
round-8 (d5253b4) corrected `codec.rs` CTRL_CHAR / GR_CHAR limit decode
(`(data[off] as i8) as f64`), and round-8-followup (674dd37) corrected the
bridge qsrv pvByte mapping. One remaining sign-extension hole survived in
`crates/epics-base-rs/src/types/value.rs::EpicsValue::to_f64` for the
`Char(u8)` arm, which widened the byte unsigned (`*v as f64`), so a record
holding 0xFF was reported as 255.0 instead of -1.0 by every numeric
consumer (db_get conversions, format helpers, etc.). Fixed by
reinterpreting the storage byte as `i8` before widening:
`Some((*v as i8) as f64)`. CharArray storage stays `Vec<u8>` because that
matches the CA wire byte pattern; only the f64 promotion needed the cast.
`cargo check -p epics-ca-rs` clean. The C-side audit targets `com_buf.rs`
and `com_que_recv.rs` do not exist (no comBuf-style wire-buffer abstraction
in ca-rs — bytes flow through tokio reads/writes directly), so those targets
are N/A.
