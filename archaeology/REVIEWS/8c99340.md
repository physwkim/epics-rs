C ref: `modules/ca/src/client/CAref.html` documentation clarification stating that `ca_create_subscription(..., COUNT=0, ...)` requests a variable-size array subscription where the server reports the current element count on each update. ca-rs's subscription API (`CaChannel::subscribe` / `CaChannel::subscribe_with_deadband` at `crates/epics-ca-rs/src/client/mod.rs:1005-1039`) does not currently expose `count` to callers — the coordinator at `mod.rs:1314-1315` always uses the channel's `native_type` and `element_count` snapshot when issuing `EVENT_ADD`. So count=0 is not currently expressible by an API user, which means this finding describes a missing-feature gap rather than a present bug. Crucially, the *receive* side already does the right thing: `SubscriptionRegistry::on_monitor_data` at `crates/epics-ca-rs/src/client/subscription.rs:65-119` decodes each update with the `count` field from the response header (parameter `count: u32`, used as `count as usize` in `EpicsValue::from_bytes_array`) — it does not truncate to a stored initial element count. So if/when a `subscribe_with_count(count: u32)` API is added, varying-size monitor updates will already be handled correctly with no extra work. No bug to fix today; doc only.
