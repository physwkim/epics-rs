The audit target `src/server/database/db_ca.rs::dbca_task` does not exist in `epics-base-rs`. There is no `db_ca.rs`, no `dbCaTask` worker, and no CA-link-driven `db_process(prec)` path that could omit `dbScanLock`. CA-link semantics in base-rs are served via `LinkSet`/`ExternalPvResolver` (synchronous get/put through `PvDatabase::resolve_external_pv`), and any record processing triggered by a CA monitor update flows through `PvDatabase::process_record_with_links`, which acquires `rec.write().await` — the analog of `dbScanLock` — for the entire processing transaction. There is no separate CA-link task that calls a record-processing entry point without holding the record lock, so the missing-scanlock race the C fix patches has no counterpart here.
