The C bug was `histogramRecord::wdogCallback` passing `prec->bptr` (raw array buffer pointer) to `db_post_events` instead of `&prec->val` (field address). `db_post_events` uses the field address as an identity token; passing the data pointer corrupted offset arithmetic. In `epics-base-rs`, `server/records/histogram.rs::HistogramRecord` owns its `val: Vec<i32>` directly and there is no analogous "watchdog callback" — `process()` mutates `val` in place and the framework dispatches monitor events via `RecordInstance::notify_field("VAL", ...)` (record_instance.rs:1426), which keys subscribers by **field name string**, not by raw pointer. Rust's owned `Vec<i32>` plus name-based subscriber dispatch makes the C wrong-pointer bug unrepresentable. No `bptr` field, no `db_post_events(prec, ptr, mask)` API, no field-address arithmetic — structurally absent.
