# Review: 5ba8080

## Verdict
not-applicable — structurally absent in base-rs.

## Analysis
The C bug is about ordering of `db_post_events(NORD)` (called from
`readValue()`) before `recGblGetTimeStampSimm()` was invoked in
`waveformRecord::process()`, so the very first monitor update for
NORD carried a zero/epoch timestamp.

`epics-base-rs/src/server/records/waveform.rs` does not implement a
`process()` override; the trait default in
`server/record/record_trait.rs:164` is used. NORD is not posted from
inside any device-support read in this rewrite — it is updated as a
side effect of `WaveformRecord::put_field("VAL", ...)`
(`waveform.rs:226-258`) and surfaced through the snapshot.

The orchestration in `RecordInstance::process_local`
(`record_instance.rs:1134-1311`) sets `self.common.time =
get_current()` (line 1248) BEFORE building `changed_fields` and
notifying subscribers (line 1269+). Subscribers are notified via
`notify_from_snapshot` (line 1397) which uses the per-record snapshot
already stamped with the current timestamp. There is no "post inside
read, stamp later" ordering hazard.
