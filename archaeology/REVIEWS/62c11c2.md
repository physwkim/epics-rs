The C bug is `dbDbLink::processTarget()` setting `pdst->rpro=TRUE` for self-links (psrc==pdst), causing an infinite async-reprocess loop on records like `chain5_cnt` (calcout.OUT="self.A PP"). base-rs has no `process_target` / `processTarget` analog: link writes go through `PvDatabase::write_db_link_value` (database/links.rs:122), which calls `put_pv` and, for PP+Passive targets, recursively invokes `process_record_with_links` with the existing `visited` HashSet — guaranteed self-link termination because the source record is already in `visited`. A grep for assignments to `common.rpro` shows the field is only set externally (CA/iocsh writes to `RPRO`); no internal code path sets RPRO during a put-while-async-pending sequence. The C-style PUTF/RPRO/procThread machinery that produces this bug simply does not exist in base-rs, replaced by visited-set + depth-limit cycle protection in `process_record_with_links_inner` (processing.rs:35). No fix needed.
