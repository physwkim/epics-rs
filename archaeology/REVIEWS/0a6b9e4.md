The C bug was `dbScan.c::scanStop` checking only `scanCtl == ctlExit` for early return; if called from an `iocInit` failure path before `scanStart`, it tried to signal events on never-spawned scan threads (deadlock/crash). base-rs `crates/epics-base-rs/src/server/scan.rs::ScanScheduler` has no `scan_stop` / `stop()` method at all — `run_with_hooks` runs indefinitely, with task lifetime tied to the scheduler's owning future. There is no `ctlInit`/`ctlRun`/`ctlExit` state machine to mishandle and no Option<JoinHandle> to unwrap on a not-yet-started task. A grep across the workspace for `scan_stop`, `ScanState`, `ctlInit` finds zero hits in source (only archaeology metadata). Closing as not-applicable; if a future commit adds graceful-shutdown semantics for the scan scheduler, this finding should be revisited then.
