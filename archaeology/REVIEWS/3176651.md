The C bug was that `dbGet` dereferenced `pfield` at offset 0 when `no_elements == 0`, reading past array bounds via raw-pointer arithmetic. In `epics-base-rs`, the equivalent path is `field_io.rs::get_pv` which returns the `EpicsValue` directly via `Record::get_field`. The value is an owned `EpicsValue::*Array(Vec<T>)` — empty arrays are `Vec::new()`, and any subsequent slice indexing is bounds-checked at runtime by Rust. There is no raw-pointer-plus-offset path that could read past element 0; an empty `DoubleArray` is returned as-is. The behavioral question (a downstream caller invoking `to_f64()` on an empty array gets `None` → `unwrap_or(0.0)` returns 0.0) is shared by the value layer rather than the get path, and `to_f64` already returns `None` for every array variant — distinguishing "empty array" from "nonempty array" at scalar-extraction time would be a wider behavior change beyond this commit's scope. The specific OOB-memory-read bug from the C fix cannot occur.
