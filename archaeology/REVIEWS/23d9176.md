The C cleanup commit had two parts: (1) `waveformRecord::process` was discarding the return value of `readValue()` and always returning 0, masking device errors; (2) `aaiRecord::init_record` had a verbose `nord` initialization and `waveformRecord::readValue` used a too-broad `!dbLinkIsConstant(&prec->siol)` guard for NORD-change event posting (replaced by `nRequest != prec->nord`). The audit targets do not exist in `epics-base-rs`: there is no `aai.rs` record at all, and `src/server/records/waveform.rs` does not implement `Record::process` (it relies on the trait default which returns `Ok(ProcessOutcome::complete())`). There is no `read_value`/`readValue` function anywhere in the crate, and no `siol`/simulation read path on waveform. NORD updates in Rust are computed directly inside `WaveformRecord::put_field` (`waveform.rs:230-253`) from the actual incoming array length (`self.nord = arr.len() as i32`) — already the correct "compare actual count" semantics the C fix adopted. Because waveform `process` never calls a fallible `read_value`, there is no return-status-swallowing bug to fix. Both audit issues are structurally absent.
