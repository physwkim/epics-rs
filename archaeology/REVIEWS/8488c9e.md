# Review: 8488c9e — initHookName STATIC_ASSERT

## Verdict
not-applicable — the C bug is `initHooks.c::initHookName` indexing a
`stateName[]` table by `initHookState` enum value with only a runtime
bounds check; no `STATIC_ASSERT(NELEMENTS(stateName) ==
initHookAtEnd+1)` enforced enum/array consistency at build time.

base-rs has no `initHookName()` analog. The init-hook surface is
captured in `crates/epics-base-rs/src/server/ioc_app.rs` as a
`Vec<Box<dyn FnOnce() + Send>>` (`after_init_hooks`, lines 61, 80,
98, 159, 261, 401) — a closure list, not an enum-indexed name table.
There is no `InitHookState` enum, no `stateName[]` slice, and no
numeric-index → string lookup path. Adding a new "init hook stage"
in Rust would mean pushing another closure or extending the small
ad-hoc API, neither of which has the array/enum-desync hazard the
C `STATIC_ASSERT` was guarding.

## C reference
`modules/libcom/src/iocsh/initHooks.c:initHookName` — added
`STATIC_ASSERT(NELEMENTS(stateName)==initHookAtEnd+1)` immediately
before the runtime bounds check.
