The C bug is in a test framework that does not exist in base-rs.
Per-test NaN comparisons in any future Rust tests would naturally use
`f64::is_nan()` / `assert!(v.is_nan())` rather than `assert_eq!`,
since the Rust `f64::NAN == f64::NAN` is `false` by language design
and `clippy::float_cmp` would catch a naive equality on most call
sites.

## Decision
**not-applicable** — the offending macro and the `dbrType→name`
formatting concern have no structural counterpart in base-rs. The
audit recommendation should be revisited when a Rust test-helper
crate analogous to `dbUnitTest` is introduced.
