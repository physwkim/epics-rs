The `asUseIP=1` semantics from the C commit are already the default in epics-rs. `epics-ca-rs/src/server/tcp.rs:591` sets `state.hostname = peer.ip().to_string()` from the verified `accept(2)` peer address; CA_PROTO_HOST_NAME is only honored when `EPICS_CAS_USE_HOST_NAMES=YES` (line 814) and even then must pass forward-DNS verification (`host_resolves_to_peer`, line 830) — strictly safer than C rsrv. Access-security HAG matching in `epics-base-rs/src/server/access_security.rs:102-111` compares the resolved hostname string to HAG members, so an operator who writes IPv4 strings in `HAG(...)` with the default config gets exactly `asUseIP=1` behaviour. The remaining gap (resolving hostname HAG entries to IPs at ACF parse time so `USE_HOST_NAMES=NO` ACFs can mix names + IPs) is a feature, not a security bug, and would change ACF semantics for existing users — out of scope.
