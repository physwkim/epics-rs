# Review: 3b3261c

## Verdict
not-applicable — structurally absent in base-rs.

## Analysis
The C bug is that a previous commit had introduced a new
`S_db_emptyArray` status returned by `dbGet`/`dbDbGetValue` for
zero-element arrays. The revert restores `S_db_badField` so existing
callers checking that specific status keep working.

`epics-base-rs` has no `S_db_emptyArray`-equivalent variant on
`CaError` (`crates/epics-base-rs/src/error.rs`). The CaError enum
contains `FieldNotFound`, `ReadOnlyField`, `TypeMismatch`,
`InvalidValue`, `PutDisabled`, etc.; no `EmptyArray`/`BadField` split.

The three call sites that branch on `value.is_empty_array()`
(`server/database/field_io.rs:82, 210, 379`) all live on the PUT side
and reject empty-array → scalar coercion with
`CaError::InvalidValue("empty array cannot be coerced to scalar field
…")`. They cite the upstream C fix (12cfd41) for silent-zero
coercion, which is a different bug from this commit's revert.

There is no `db_get`/`dbGet` analog in `epics-base-rs` that would
return a distinct empty-array status to be reverted; reads go through
`record.get_field` which returns `Option<EpicsValue>` with no
empty-array-specific error path. The wire-protocol-compat hazard the
revert protects (callers matching on a specific status code) cannot
exist here because no such status was ever added.
