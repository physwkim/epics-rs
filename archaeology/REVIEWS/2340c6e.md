The C bug was about `paddr->pfield = prec->bptr` cached at link-resolution time (`cvt_dbaddr`) when the buffer was still `NULL` — a follow-up read produced a NULL deref. In `epics-base-rs`, the compress/histogram/subArray records own their data buffer directly inside the struct (`compress.rs::CompressRecord::val: Vec<f64>`, similarly for histogram and asub_record). Field access goes through the `Record::get_field`/`put_field` trait at access time (`record.get_field("VAL")` returns the live `EpicsValue` clone), not via a cached raw pointer. Rust's owned `Vec<T>` is always non-null and length-tracked, so the NULL-`bptr` and wrong-field-address bugs are structurally eliminated. There is no `cvt_dbaddr` / `dbAddr` mechanism in the Rust database — the field-name match runs every access and reads the current owner value.
