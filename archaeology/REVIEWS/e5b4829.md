The C bug was `lsiRecord.c` and `lsoRecord.c` failing to cap `SIZV` at `0x7FFF` before assignment to `dbAddr::field_size` (a signed `short`), causing wrap-around to negative values for `SIZV > 32767`. In `epics-base-rs`, the `lsi`/`lso` (long-string-input/output) record types are not implemented — `server/records/` contains `stringin.rs` / `stringout.rs` (40-byte fixed-size string records) but no long-string variants, no `SIZV` field, and no `field_size` analogue. The Rust framework dispatches strings as `EpicsValue::String(String)` whose length is a `usize` (machine-word) tracked by Rust's owned String — there is no signed-short boundary to wrap. The CA wire-protocol path serializes string length as `u32` in the DBR header. Structurally absent: no records to fix, no signed-overflow surface to guard.
