The C bug was in `std/filters/ts.c`: the timestamp filter called `cantProceed()` (which calls `abort()` and crashes the IOC process) on recoverable conditions — `tsModeInvalid` / `tsModeGenerate` in `filter()` and `tsStringInvalid` in `ts_string()` — and the value-callback signature was `void`, so `replace_fl_value` could not detect/propagate failure. The fix changed callbacks to return `int`, replaced `cantProceed` with `errMessage` + NULL `pfl`, and moved the `tsStringInvalid` guard before the malloc. The audit target `crates/epics-base-rs/src/server/database/filters/ts.rs` does not exist; in fact the entire `filters/` directory does not exist (`crates/epics-base-rs/src/server/database/` contains only `db_access.rs`, `field_io.rs`, `link_set.rs`, `links.rs`, `mod.rs`, `processing.rs`, `scan_index.rs`). There is no Channel Access server-side filter framework (`{ts}`, `{dbnd}`, `{arr}`, `{sync}` JSON-filter syntax) implemented in base-rs at all — no `FieldLog` type, no `replace_fl_value`-equivalent, no `channelRegisterPost`. A workspace grep for `tsMode`, `field_log`, `FieldLog`, `replace_fl_value`, and `cantProceed` finds zero hits in src. All `panic!` / `unreachable!` occurrences in `crates/epics-base-rs/src/server/` are inside `#[cfg(test)]` test bodies, not production logic-error paths. Closing as not-applicable; if a future commit adds a server-side filter pipeline, the audit recommendation that filter trait methods return `Result<Option<FieldLog>>` (rather than panicking on invalid enum variants) should be applied at that time.
