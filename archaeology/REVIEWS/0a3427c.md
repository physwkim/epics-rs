The C bug was `libcom/log/logClient.c::logClientClose` resetting `nextMsgIndex = 0` and `memset`-ing `msgBuf` on TCP disconnect, silently discarding buffered-but-unsent log messages across a reconnect. In `epics-base-rs` there is no `iocLogClient` / `log_client` subsystem at all — `runtime/log.rs` is a four-macro shim (`rt_debug!`, `rt_info!`, `rt_warn!`, `rt_error!`) that delegates to `eprintln!` (sync stderr write). There is no TCP log-collector client, no buffered queue, no reconnect loop. Operators relying on iocLogServer collection use the `tracing` crate ecosystem (subscribers, layers, OTLP exporters) configured at the application level, not a base-rs internal client. Structurally absent.
