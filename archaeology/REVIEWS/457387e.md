C bug was in the `dbf_type_to_text(type)` macro in `db_access.h`, which used `((type) >= -1 && (type) < dbf_text_dim-2)` to bounds-check before indexing the static `dbf_text[]` array. Passing an unsigned `type` triggered `-Wtype-limits` and, worse, `(unsigned)-1 == UINT_MAX` would silently pass the lower bound and read past the array. ca-rs has no equivalent macro and no `dbf_text[]` lookup table — the closest analogue is `DbFieldType::from_u16(v: u16) -> CaResult<Self>` at `crates/epics-base-rs/src/types/dbr.rs:32`, which is a `match` on the seven valid values (0..=6) returning `Err(CaError::UnsupportedType(v))` for everything else. There is no array indexing; out-of-range codes are converted to a typed error rather than a buffer-relative read. The `Display`/`Debug` impls on `DbFieldType` are derived from the enum itself, so a malformed code never reaches a stringification site. The class of bug (signed-vs-unsigned bounds confusion + array index UB) is structurally absent from the Rust port. No code change required.
