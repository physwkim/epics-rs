The audit target `src/server/database/dev_ai_soft.rs::read_ai` does not exist. `epics-base-rs` has no separate `Soft` device-support layer for `ai`; the `AiRecord` (`src/server/records/ai.rs`) is a single record impl with no `read_ai` device function and no per-device "return 2 = no-convert" status code.

INP/SIOL link reads happen in the processing layer (`src/server/database/processing.rs::read_input_link` family), which uses `CaResult<EpicsValue>` and propagates `Err` directly — there is no `return 2` early-success path that could mask a failed link read. Link failure naturally propagates as `Err(...)`, so the alarm/UDF path runs in the processing layer, not in a device support shim.

The C bug was specifically a control-flow ordering mistake inside `devAiSoft.c::read_ai` (`return 2` placed outside the `if(status==0)` block). With no `dev_ai_soft.rs` in the Rust tree, the bug pattern is structurally absent.
