# Review: 19146a5 — Windows SO_REUSEADDR is dangerous

## Verdict
applied — three UDP bind paths in ca-rs / epics-base-rs called
`set_reuse_address(true)` unconditionally. On Windows this allows any
process to rebind the same port and hijack/inject CA traffic.

## Fixes
1. `server/udp.rs::run_single_responder` — wrapped `set_reuse_address(true)`
   in `#[cfg(not(windows))]`. Combined with the 5064931 / 65ef6e9 fix the
   block also calls `set_reuse_port(true)` on Unix.
2. `repeater.rs::run_repeater` — added `set_reuse_address` / `set_reuse_port`
   under `#[cfg(not(windows))]` / `#[cfg(unix)]` (the in-process repeater
   previously set neither).
3. `epics-base-rs/net/async_udp_v4.rs::bind_one_at` — gated
   `set_reuse_address(true)` with `#[cfg(not(windows))]`. This is the
   shared `AsyncUdpV4` used by ca-rs client search and beacon-monitor.

## C reference
`modules/libcom/src/osi/os/default/osdSockAddrReuse.cpp:
epicsSocketEnableAddressReuseDuringTimeWaitState` — `#ifndef _WIN32`
guard around the `setsockopt(SO_REUSEADDR)` call.

## Build
`cargo check -p epics-ca-rs` — clean.
