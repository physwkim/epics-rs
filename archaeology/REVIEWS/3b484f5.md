# Review

## Audit Targets
- `src/server/database/const_link.rs::load_scalar` / `load_array` —
  no such file. base-rs handles constant links inside
  `ParsedLink::Constant` produced by
  `src/server/record/link.rs::parse_link_v2`.

## Verification
`parse_link_v2` already returns `ParsedLink::None` when the trimmed
input is empty (line 76). The remaining gap is the **quoted** empty
string `""`:

```
if link_part.starts_with('"') && link_part.ends_with('"') && link_part.len() >= 2 {
    return ParsedLink::Constant(link_part[1..link_part.len() - 1].to_string());
}
```

For `link_part = "\"\""`, `len() == 2`, so the body returned
`Constant("")` — a syntactically valid but semantically meaningless
constant. Downstream code (e.g. `constant_value` in link.rs:54-66)
would feed the empty string into `parse::<f64>()` and silently
produce `EpicsValue::String("")` rather than an error. This is the
exact behavior `3b484f5` removed in C: NULL-equivalent input
(`!pstr || !pstr[0]`) must be rejected as `S_db_badField`, not parsed
as a default value.

## Fix
After matching the `"..."` quoting, check whether the inner slice is
empty and, if so, return `ParsedLink::None` — the same outcome as the
"trim==empty" guard above. This makes `""` and the empty / missing
case indistinguishable to callers, matching dbConstLoadScalar /
dbConstLoadArray after the C patch. Numeric / non-empty-quoted
constants are unaffected.
