## Verification
- All built-in command handlers in `iocsh/commands.rs` already return
  `CommandResult` (`Result<CommandOutcome, String>`) and propagate
  parse / file-load errors via `Result::Err`. There are no `void`
  wrappers analogous to the C `db*CallFunc` shells — the trait
  `CommandHandler::call` already carries the error.
- `IocShell::execute_command_inner` (mod.rs:108) bubbles the handler
  result through `execute_command` and `execute_line` unchanged.
- The only error-swallowing site was `IocShell::execute_script`
  (mod.rs:112): it logged each per-line failure but returned
  `Ok(())` regardless, so callers (e.g., automated startup
  verification) had no way to detect that a command in the script
  had failed. This is the structural equivalent of the C bug —
  the C wrappers logged via the iocsh redirector but never set the
  shell's error code.

## Fix
`execute_script` now retains the most recent per-line error in
`last_err: Option<String>` and returns
`Err(...)` from the script call when at least one command failed.
Per-line behavior is unchanged: errors are still printed with
`<path>:<lineno>: Error: <msg>` and execution continues, exactly like
C iocsh after `144f975`. Only the final return value changes — this
is the equivalent of `iocshSetError` propagating a non-zero status to
the script's caller.

The early-exit branch (`CommandOutcome::Exit`) likewise returns the
recorded error if any.
