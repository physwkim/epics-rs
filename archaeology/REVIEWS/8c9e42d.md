The C bug was numeric overflow in `epicsStrnRawFromEscaped`: octal `\NNN` consumed a third digit even when the accumulator already exceeded `0377`, and hex `\xXX...` greedy-consumed unlimited digits — both producing values > 0xFF then storing into `char` (UB). base-rs does not implement `epicsStrnRawFromEscaped` or any analogous octal/hex escape decoder. `crates/epics-base-rs/src/server/iocsh/registry.rs` only honors `\\` and `\"` inside quoted args (no numeric escape handling at all). `crates/epics-base-rs/src/server/db_loader/mod.rs::read_quoted_string` only matches `"`, `\\`, `n` — anything else is preserved verbatim with the leading backslash. Neither path can overflow a u8 from accumulating digits. Closing as not-applicable.
