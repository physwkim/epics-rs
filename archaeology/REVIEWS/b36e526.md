# Review: b36e526

## Verdict
not-applicable — structurally absent in base-rs.

## Analysis
The C bug is in `lnkConst_loadArray()` (`modules/database/src/std/link/lnkConst.c`),
where the `sc40` scalar-string-constant case was routed through
`dbFastPutConvertRoutine[DBF_STRING][dbrType]` for DBF_CHAR waveform
targets, capping the string at the 40-byte `epicsOldString` limit
instead of using direct byte copy up to `NELM`.

`epics-base-rs` does not implement the `lnkConst` JSON-link mechanism
(`{const:"..."}`). A repo-wide search for `lnkConst`, `{const:`,
`const_link`, `Const.*Link` in `src/` returns zero hits. The link
parser (`server/record/link.rs:74-170`) recognises numeric and quoted
string constants but stores them as plain `ParsedLink::Constant(String)`
without a per-target type-conversion table; the `epicsOldString`
40-byte cap does not exist anywhere in base-rs.

Loading from `.db` files goes through `db_loader/mod.rs:436-464` →
`EpicsValue::parse(dbf_type, value_str)` (`types/value.rs:604`) →
`record.put_field`. The waveform record (`records/waveform.rs:218-260`)
coerces `EpicsValue::String → CharArray` via `s.as_bytes().to_vec()`
with full string length (up to NELM via `arr.resize(nelm, 0)`). There
is no 40-byte truncation path.
