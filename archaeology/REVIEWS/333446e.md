# Review: 333446e — lockset ownership debug assert

## Verdict
not-applicable — the C bug is `dbDbLink.c::processTarget` calling
`dbPutLink` without verifying the calling thread holds the lockset
for both source and destination records under `LOCKSET_DEBUG`. The
fix adds `assert(psrc->lset->owner == self && pdst->lset->owner ==
self)` and a missing `epicsThread.h` include.

base-rs has no lockset-with-owner-thread machinery: no `LockSet`,
`scan_lock`, or `lockset` types exist anywhere in the crate (verified
by grep). Per-record concurrency uses Tokio `RwLock<RecordInstance>`
from `crates/epics-base-rs/src/server/database/links.rs`. Recursive
record processing through DB links uses `write_db_link_value()`
(`links.rs:122-146`) followed by `process_record_with_links()`, which
relies on `RwLock` rather than the C lockset graph. The debug
assertion has no analog because Rust's `RwLock` already enforces
exclusive `write()` ownership statically through the borrow checker,
and there is no separate "lockset owner field" to test against.

## C reference
`modules/database/src/ioc/db/dbDbLink.c:processTarget` — added
`#ifdef LOCKSET_DEBUG` ownership assertions on both source and dest.

No code changes; `cargo check -p epics-base-rs` already clean.
