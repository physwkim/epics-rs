The C bug was in `dbCa.c::dbCaGetLink`, which is the CA-link integration layer (a separate translation between the upstream CA client and the local in-process database). In `epics-base-rs`, CA-link reads are routed through `PvDatabase::resolve_external_pv` (`src/server/database/mod.rs:287`) and `read_link_value` (`src/server/database/links.rs:19`) which return `Option<EpicsValue>`. Empty arrays are preserved as typed `*Array(Vec::new())` (already enforced by P-1 / commit `8cc2039` in `EpicsValue::from_bytes_array`). There is no fast-buffer-deref path that could read past element 0 — Rust slice indexing is bounds-checked and the in-process value is owned, not a raw pointer to a CA buffer. The full CA-link client logic lives in the `epics-ca-rs` crate, not `epics-base-rs`.
