The C bug exists in `dbCa.c::testdbCaWaitForEvent` and `dbCa.c::dbCaSync` — neither has a Rust counterpart. There is no `db_ca_sync`, `testdb_ca_wait_for_event`, or `db_ca` module anywhere under `crates/`. Same gap as 717d69e: the dbCa worker subsystem (CA-link integration in the IOC core, plus its test sync helpers) has not been ported. When that subsystem is built, the test-helper sync path must (a) flush the dbCa worker queue, (b) flush the CA client context's I/O queue, and (c) ensure callbacks holding the wait state have returned before the state is dropped — `Arc<Mutex<WaitState>>` semantics with a `Weak` callback handle naturally prevent use-after-free, but the ordering between worker-flush, context-flush and state-drop must be explicit. No fix to apply today.
