# Review

## Audit
The C bug is `nciu::write()` rejecting `countIn == 0` with `cacChannel::outOfBounds`,
preventing legitimate zero-element waveform puts.

In `epics-ca-rs`, the put surface lives on `CaChannel`:

- `put`              (`crates/epics-ca-rs/src/client/mod.rs:880`)
- `put_with_timeout` (`crates/epics-ca-rs/src/client/mod.rs:926`)
- `put_nowait`       (`crates/epics-ca-rs/src/client/mod.rs:970`)

All three pull `count = value.count() as u32` from `EpicsValue`
(`crates/epics-base-rs/src/types/value.rs:398`) and forward it to
`TransportCommand::Write{,Notify}` without any `count == 0` rejection or
upper-bound (`count > native`) guard. `EpicsValue::count()` returns `0` for
empty array variants and the put pipeline accepts that.

The transport encoder at `crates/epics-ca-rs/src/client/transport.rs:199-240`
also accepts `count == 0` and the protocol header builder
(`crates/epics-ca-rs/src/protocol.rs:278`) emits the compact form
(`postsize=0, count=0`) for that case — distinct from the extended-form
sentinel (`postsize=0xFFFF && count==0 && extended_postsize.is_some()`).

Empty-array decoding is already handled at
`crates/epics-base-rs/src/types/value.rs:267` (`from_bytes_array`) and is
explicitly cited there as the round-trip partner for empty puts.

## Verdict
not-applicable — the bug class is structurally absent. The Rust put path
never reproduced the libca `countIn == 0` rejection.
