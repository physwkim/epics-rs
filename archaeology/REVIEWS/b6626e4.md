The C bug is a missing diagnostic in `dbEvent.c::event_read`: when the drain loop exits with the last callback notified `eventsRemaining=1` but the consumer did not flush further, no warning was logged. The fix adds a `possibleStall` flag and a one-shot `errlogPrintf("dbEvent possible queue stall")` line.

`crates/epics-base-rs` has no `event.rs`, no `event_que`/`event_read`/`eventsRemaining`/`possibleStall` analogues, and no per-channel ring-buffered event queue. Subscription/monitor dispatch in base-rs is handled via tokio channels in `epics-ca-rs` and `epics-pva-rs`; there is no in-process db-event drain loop here that would produce the C-style stall pattern. Backpressure on a tokio bounded channel surfaces directly as `try_send` `Err(Full)` or `send().await` blocking — the C-level stall flag has no place to attach. The defect is structurally absent in this crate.
