The C bug was that `compressRecord::special` handled `SPC_RESET` (write to `RES`) by calling `reset(prec)` but skipped the subsequent `monitor(prec)` call, so CA monitor subscribers received no notification of the cleared buffer. In `epics-base-rs`, the compress record (`src/server/records/compress.rs`) does not implement `special()`. Instead, the reset is performed inside `Record::process()` (lines 130-140) when `self.res != 0`, which is invoked unconditionally by `put_record_field_from_ca` after every CA put (`src/server/database/field_io.rs:502`). Once `process()` returns, the snapshot pipeline in `process_record_with_links_inner` (`src/server/database/processing.rs:674-727`) posts `VAL` (always, when `include_val`) and any auxiliary subscribed fields whose value changed (e.g., `NUSE`, lines 682-707). The C `monitor(prec)` posts exactly `NUSE` and `VAL` with `DBE_VALUE|DBE_LOG`; the Rust pipeline covers both. The architecture difference (process-driven monitor pipeline vs. C `db_post_events` from inside `special`) makes the C-shaped bug structurally absent.
