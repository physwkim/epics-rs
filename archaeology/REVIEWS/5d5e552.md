The C commit added `initHookAnnounce` calls at each phase of `iocShutdown` (`initHookAtShutdown`, `initHookAfterCloseLinks`, `initHookAfterStopScan`, `initHookAfterStopCallback`, `initHookAfterStopLinks`, `initHookBeforeFree`, `initHookAfterShutdown`) so third-party code registered via `initHookRegister` can synchronize teardown with scan/callback/dbCa cleanup.

`crates/epics-base-rs` does not implement an `ioc_shutdown` sequence. Grep for `ioc_shutdown|iocShutdown|shutdown\(|stop\(|abort` in `server/ioc_app.rs` returns no matches — `IocApplication` only exposes pre-init / post-init hook lists (`after_init_hooks`) and relies on the tokio runtime dropping when `run` exits. There is no `dbCaShutdown`, no `scanCleanup`, no `callbackCleanup`, no `iterateRecords(doCloseLinks)` analogue, and no `initHook` enum to extend. Structured shutdown phases would be a forward-looking design item, not a fix to existing code; the C bug is not present because the entire shutdown subsystem is structurally absent in this crate.
