The C bug was that `longoutRecord::special` set `prec->outpvt = OUT_LINK_CHANGED` in the pre-call (`!after`) branch instead of the post-call (`after`) branch, creating a race where unrelated processing could consume the flag before the OUT-link update completed. The audit target `src/server/records/longout.rs::special` does not exist — `LongoutRecord` is a `#[derive(EpicsRecord)]` declarative struct (`src/server/records/longout.rs`) with no `special()` override, no `outpvt`/`OUT_LINK_CHANGED` field, and no `OOPT`/`OOCH` fields (those live on `calcout`/`scalcout` only). The Rust longout has no force-write-on-link-change feature, so the pre/post ordering bug has no analog here. The default `Record::special` (`src/server/record/record_trait.rs:278`) is a no-op. If/when a future longout extension adds a force-write flag, the convention from `field_io.rs:407` and `:419` is that `special(field, false)` runs *before* `put_field` and `special(field, true)` runs *after* — placing flag mutation in the `after == true` branch is the natural choice.
