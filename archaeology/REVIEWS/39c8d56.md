The C bug had two parts: (a) `dbGet` operating on a post-filter field-log with `no_elements=0` would memcpy/size-calc a zero-length buffer through raw pointer arithmetic; (b) `dbDbGetValue` running filter chains on empty channels. Neither pattern exists in `epics-base-rs`. The Rust `get_pv` returns an owned `EpicsValue` whose array variants are `Vec<T>` — empty means `Vec::new()`, with no associated raw pointer or memcpy. Bounds checks are unconditional. The Rust DB-link layer (`links.rs`) does not currently implement filter chains, so the second crash mode is structurally absent. The spurious-warning init-time issue is also absent: `register_cp_link` / `setup_cp_links` log only when CP links exist; no per-record zero-element warning fires at IOC init.
