The C bug is `dbScan.c::scanStop` returning after the stop event was acknowledged but BEFORE the periodic / once scan threads were joined (`joinable=0`, no `epicsThreadMustJoin`), letting threads keep touching the database while `scanCleanup` freed it — a classic UAF. The audit target `src/server/database/scan.rs::scan_stop` does not exist in base-rs: scan logic lives in `crates/epics-base-rs/src/server/scan.rs::ScanScheduler` (and `scan_event.rs::ScanSchedulerV2`) and neither type exposes any `stop` / `scan_stop` / `shutdown` method — `run_with_hooks` is intended to run for the lifetime of the IOC, and tasks are spawned via `crate::runtime::task::spawn` returning `JoinHandle`s that loop forever. Database access from those tasks goes through `Arc<PvDatabase>`, so even when the parent future is dropped, no UAF is possible: the Arc keeps the DB alive until every task drops its clone. The C precondition (joinable=0 detached threads racing with manual free) cannot reproduce in safe Rust here. Note: the current `run_with_hooks` does drop 6 of 7 periodic-scan handles (only `handles.into_iter().next()` is awaited) — a stylistic concern but not the audited bug, since dropping a `JoinHandle` does not cancel its task. Closing as not-applicable; revisit if graceful-shutdown semantics are added to ScanScheduler.
