The C bug was in the `dbnd` channel filter: `send = 0` was assigned unconditionally before `recGblCheckDeadband`, so when the value sat inside the deadband, ALARM and PROPERTY mask bits were also dropped from the posting mask. base-rs has no channel-filter pipeline (no `filters/dbnd.rs` or equivalent). The structurally analogous code in `crates/epics-base-rs/src/server/database/processing.rs:653-672` already handles the three bit groups independently — `EventMask::VALUE`/`LOG` are gated on `check_deadband_ext()` while `EventMask::ALARM` is OR'd in unconditionally on `alarm_result.alarm_changed`, and the dispatcher (`record_instance.rs::notify_from_snapshot`) routes per subscriber via `intersects(posting_mask)`. PROPERTY is defined in `recgbl::EventMask::PROPERTY` (0x08) but no posting path emits it yet — that is a separate gap, not the C dbnd suppression bug. No code change applied.
