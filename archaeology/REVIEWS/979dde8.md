The C bug was `dbAccess.c::get_enum_strs` walking enum slot strings via raw pointer arithmetic (`ptemp += sizeof(strs[0])`) which hid per-slot size from `__builtin_object_size` and tripped `_FORTIFY_SOURCE=3` warnings on `strncpy`; the buffer-pointer advance was also mispositioned, allowing double-advance on early-exit paths. In `epics-base-rs`, the CA DBR enum-strings encoder is `types/codec.rs::encode_enum_metadata` (lines 403–422). It uses **indexed slice access** throughout:

```
for i in 0..MAX_ENUM_STATES {
    let mut slot = [0u8; MAX_ENUM_STRING_SIZE];
    if let Some(s) = ei.strings.get(i) {
        let bytes = s.as_bytes();
        let len = bytes.len().min(MAX_ENUM_STRING_SIZE - 1);
        slot[..len].copy_from_slice(&bytes[..len]);
    }
    buf.extend_from_slice(&slot);
}
```

No raw pointer arithmetic, no `strncpy`, slice bounds checks are mandatory at the language level. The buffer is unconditionally `extend_from_slice`d every iteration, and the `else` branch (no enum info) writes the same total `MAX_ENUM_STATES * MAX_ENUM_STRING_SIZE` zero-fill — so there is no early-exit path that can desynchronize the cursor. Both the FORTIFY surface and the buffer-advance bug are structurally absent.
