The C fix sets `S_db_badChoice` and calls `dbMsgPrint` to attach
"using menu <name>" / "no such device support for '<rec>' record type"
context in `dbPutStringNum` (modules/database/src/ioc/dbStatic/dbStaticRun.c)
when an integer-or-choice parse fails on a DBF_MENU/DBF_DEVICE field.

base-rs has no `dbStaticRun`/`dbStaticLib` analog. The audit target
`src/server/database/static_run.rs::put_string_num` does not exist. The
functional analog of `dbPutStringNum` is `EpicsValue::parse` in
`crates/epics-base-rs/src/types/value.rs:611`, which dispatches by
`DbFieldType` (Short/Float/Enum/Char/Long/Double/String) and on Enum/Short
falls back to `resolve_menu_string` (a hardcoded small set of EPICS menus,
not a DBD-driven menu/device-type registry). DBF_DEVICE is not modeled at
all — there is no per-record device-support choice lookup that could even
fail with "no such device support".

On a parse miss `EpicsValue::parse` already returns
`CaError::InvalidValue("invalid enum or menu string: {s}")` — the bad
input is included verbatim. The C improvement (mention which menu name,
or which record type's device-support list) requires a DBD-loaded menu
table keyed by field-name → menu-name and a `recDeviceTable` lookup;
neither structure exists in base-rs. Adding contextual menu/device names
without those tables would just hardcode strings, which is not the C fix.

Pure UX (rust_verdict: partial / severity: low) on a layer base-rs has
not implemented. No correctness/safety bug. Not applicable.
