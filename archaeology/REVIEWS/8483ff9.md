# Review: 8483ff9 — recGblResetAlarms NAMSG clear

## Verdict
not-applicable — the C bug is `recGbl.c::recGblResetAlarms` copying
`namsg` to `amsg` but not zeroing `namsg` afterwards, leaving a stale
pending-alarm message that can spuriously trigger a later `DBE_ALARM`
event when `namsg != amsg` is re-checked.

base-rs has no AMSG/NAMSG fields. `CommonFields`
(`crates/epics-base-rs/src/server/record/common_fields.rs:8`) carries
only `sevr`/`stat`/`nsev`/`nsta` plus `acks`/`ackt`; `desc` is the
only string-shaped alarm-adjacent field and is unrelated to the
alarm-message promote/reset cycle. `rec_gbl_reset_alarms()`
(`src/server/recgbl.rs:96`) already mirrors the C clear-after-promote
pattern for `nsev`/`nsta`, and there is no message field to extend
the same pattern to. Grep across the entire crate confirms zero
occurrences of `amsg`, `namsg`, `AMSG`, `NAMSG` (case-insensitive).

If/when AMSG support is later added, the corresponding pending
buffer must be zeroed in `rec_gbl_reset_alarms` next to the existing
`nsev`/`nsta` resets — pre-emptive note for that future port.

## C reference
`modules/database/src/ioc/db/recGbl.c:recGblResetAlarms` — added
`pdbc->namsg[0] = '\0'` immediately after copying NAMSG into AMSG.

No code changes; `cargo check -p epics-base-rs` already clean.
