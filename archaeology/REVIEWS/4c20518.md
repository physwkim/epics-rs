The C bug was in `recGbl.c::recGblRecordError`: it guarded the `errSymLookup(status, ...)` call with `if (status)` instead of `if (status > 0)`, so negative EPICS status codes were passed to a lookup that only handles positive codes, producing garbled error text. In `epics-base-rs` the audit target `src/server/database/recgbl.rs` does not exist; the `recgbl` module lives at `src/server/recgbl.rs` and contains only alarm-state helpers (`rec_gbl_set_sevr`, `rec_gbl_reset_alarms`, `rec_gbl_check_udf`) plus alarm-status constants and `EventMask` — no `record_error` / `errSymLookup` analogue. `Grep "record_error|errSymLookup|recgbl_record_error|RecordError"` over `src/server/recgbl.rs` returns no matches, and a crate-wide search yields no numeric-status → string conversion that could mis-handle negative codes. Error reporting in base-rs uses `Result<T, CaError>` with `thiserror`-style `Display` impls, which render messages by enum variant rather than by an integer status table, so the sign-check bug class is structurally absent.
