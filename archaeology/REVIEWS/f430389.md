The C bug is `iocInit.c::iocShutdown` gating `scanStop()` and `callbackStop()` behind `if (iocBuildMode == buildIsolated)`, so in the normal `buildFull` path the scan/callback worker threads were never told to stop and never joined — UAF on shutdown. The audit target `src/server/database/ioc_init.rs::ioc_shutdown` does not exist in base-rs: there is no `ioc_shutdown` (or any equivalent) function anywhere in the crate. `crates/epics-base-rs/src/server/ioc_app.rs::IocApplication::run` initialises everything and then hands off to a protocol runner (`protocol_runner(config).await`); the lifecycle ends when that future returns or the process exits, with no two-phase teardown. There is also no `iocBuildMode` / `buildIsolated` distinction — one code path covers both cases. Background scan / callback tasks hold `Arc<PvDatabase>` clones, so even ungraceful shutdown cannot UAF the database the way the C version did with malloc/free. The bug pattern (a conditional that skips thread-join) cannot reproduce because the conditional itself does not exist. Closing as not-applicable; if/when a graceful `ioc_shutdown` is introduced, the audit must verify scan + callback abort+await is unconditional.
