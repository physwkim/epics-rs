# Review: 275c4c7 — dbGet empty-array guard variable

## Verdict
not-applicable — the C bug is `dbAccess.c::dbGet` checking
`pfl->no_elements < 1` instead of the locally-computed `no_elements`
after filter-chain application; the filter-log struct field could
disagree with the post-filter element count and let zero-element
processing slip through.

base-rs has no `dbGet`-with-field-log path. The crate-wide
`db_access.rs` (`crates/epics-base-rs/src/server/database/db_access.rs`)
is a high-level channel abstraction (`DbChannel`, `DbSubscription`,
`DbMultiMonitor`) whose accessors return owned `EpicsValue` or scalar
extracts (`get_f64`, `get_i32`, etc.). There is no `pfl` /
`db_field_log` filter intermediate, no array-length local that could
diverge from a struct field, and no empty-array early-return guard
to mis-bind. EPICS array values are owned `Vec<T>` inside
`EpicsValue::*Array`, so length is read from `Vec::len()` only — no
duplicate `no_elements` source exists.

## C reference
`modules/database/src/ioc/db/dbAccess.c:dbGet` — replaced
`pfl->no_elements` with the local `no_elements` in the empty-array
early-exit guard.
