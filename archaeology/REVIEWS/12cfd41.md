Genuine bug. The C `dbPut` fix raised `LINK_ALARM/INVALID_ALARM` when a `nRequest=0` array was put into a scalar field. In Rust, the equivalent path was `field_io.rs::put_pv` (and siblings `put_pv_and_post_with_origin`, `put_record_field_from_ca`) which call `value.convert_to(target)` for cross-type coercion. `convert_to` invokes `to_f64().unwrap_or(0.0)`, and `to_f64()` returns `None` for any `*Array` variant — so an empty `LongArray` being put into a `Double` field would silently coerce to `Double(0.0)` and store as zero. Same-type empty arrays already failed via the macro-generated `put_field` `TypeMismatch` check, but mixed-type empty arrays slipped through. Added `EpicsValue::is_empty_array()` helper and three guards (one per put entry point) that return `CaError::InvalidValue` before the `convert_to` call, mirroring the C `LINK_ALARM` rejection. `cargo check -p epics-base-rs` passes.
