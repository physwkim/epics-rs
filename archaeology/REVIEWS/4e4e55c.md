The C bug was that `dbDeleteRecordLinks()` only freed `plink->text` instead of calling the authoritative `dbFreeLinkContents(plink)`, leaking link-type-specific allocations (PV_LINK.pvname, JSON_LINK.json.string, INST_IO.instio.string) on record deletion. The audit target `src/server/database/db_static_lib.rs::delete_record_links` does not exist in `epics-base-rs` — there is no `db_static_lib.rs`, no `delete_record_links`, and no record-deletion API at all (records are only added via `add_record`; once registered they live for the life of the `PvDatabase`). When records are dropped (e.g., when the database itself is dropped), `RecordInstance` and its owned link strings (`common.inp`, `common.out`, `common.dol`, etc.) are freed automatically by Rust's `Drop` impls. Links are stored as `String` plus a parsed `ParsedLink` enum — no untagged unions, no manual `free` per variant. The C-style memory leak from incomplete free is structurally impossible.
