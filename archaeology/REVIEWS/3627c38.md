The C bug was in `dbDbGetValue` running filter pre/post chains on a channel with `dbChannelFinalElements(chan) < 1`. In `epics-base-rs`, `links.rs::read_link_value` and `read_link_with_alarm` route DB-link reads through `get_pv`, which returns the field's `EpicsValue` directly. There is no filter-chain (`run_pre_chain` / `run_post_chain`) infrastructure in the Rust database link layer at present — DB links are pass-through reads of the source record's field. There is no read-log allocation or zero-length filter-evaluation path that could panic. If a filter mechanism is added later, the empty-channel guard (returning an error and setting LINK alarm) should be applied at that point — currently nothing to fix.
