The audit target `src/server/database/db_event.rs::db_post_events` does not exist in `epics-base-rs`. There is no `db_event.rs`, no `db_field_log` struct, and no `pre_chain`/`post_chain` filter pipeline whose mask field could be overwritten. The C bug requires a heap-allocated `pLog` whose `mask` is set once by `db_create_event_log()` and then re-used as `db_post_events()` walks the subscriber list — base-rs has no such shared mutable state.

Event posting in base-rs (see `src/server/record/record_instance.rs:1397-1456` `notify_from_snapshot` / `notify_field_with_origin`) takes the per-post mask as an argument, iterates subscribers, and tests `sub_mask.intersects(posting_mask)` at filter time. The `MonitorEvent` struct (`src/server/pv.rs:131-143`) carries only a `Snapshot` and `origin` — there is no per-event `mask` field that gets mutated and shared across iterations. Each subscriber's registered mask is read directly from its `Subscriber` row; the posting mask is a function parameter that never gets written back. No filter chain consumes a "field log mask" that could disagree with the actual posted bits. Nothing to fix.
