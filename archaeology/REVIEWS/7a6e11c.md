The C bug guards `casStatsFetch` and `casClientInitiatingCurrentThread`
against NULL `clientQlock` / NULL `rsrvCurrentClient` thread-private when
called before `caServerInit`. In epics-rs the equivalent surface is
`crates/epics-ca-rs/src/server/ca_server.rs::CaServer::stats()`, which
returns `Arc<ServerStats>` constructed unconditionally in `CaServer::new`.
Counter reads use atomic `load(Relaxed)` on always-initialised `AtomicU64`s,
so the pre-init read returns zero by construction — exactly the behaviour
the C patch installs. There is no per-thread "current client" thread-local
that would crash before init; per-connection state lives on the spawned
TCP task. The bug class cannot reproduce, so no fix is warranted.
