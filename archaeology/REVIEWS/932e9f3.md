# Review

## Verdict
**not-applicable** — base-rs access security has no DNS resolution path.

## Evidence
- `crates/epics-base-rs/src/server/access_security.rs` contains no
  `hag_add_host` and never calls `to_socket_addrs`, `lookup_host`, or any DNS
  primitive. A grep for those names plus `dns|asUseIP|asCheckClientIP` in the
  crate returns zero hits.
- `parse_acf` simply parses HAG entries as bare strings and stores them
  verbatim in `AccessSecurityConfig.hag: HashMap<String, Vec<String>>`.
  `check_access_asl` matches the client `host` parameter against those
  strings via `m == host` (case-sensitive equality on the string the caller
  supplied) — no resolution, no IP-vs-name decision, no failure path that
  could `S_asLib_noHag`-abort ACF loading.
- There is no `asCheckClientIP` toggle and no IP-form HAG handling, so the
  preconditions for the soft-fail-vs-hard-fail trade-off are absent.

## Decision
With no DNS lookup involved in ACF parsing, there is nothing to soft-fail.
If a future change adds IP-based HAG resolution (parity with C
`asHagAddHost`), apply the same pattern: log a warning and store an
`"unresolved:<host>"` sentinel that cannot accidentally match a real client,
rather than aborting the ACF load.
