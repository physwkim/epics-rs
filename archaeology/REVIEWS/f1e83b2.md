The C bug was that several output records (`aoRecord.c`, `boRecord.c`, `calcoutRecord.c`, `aaoRecord.c`, `int64outRecord.c`, `longoutRecord.c`, `mbboDirectRecord.c`, `mbboRecord.c`, `stringoutRecord.c`, `aSubRecord.c`) called `recGblGetTimeStamp()` AFTER `writeValue()`/`recGblFwdLink()`, so any downstream record with `TSEL` linked to this record's TIME read the stale pre-process timestamp. In `epics-base-rs` timestamping is not done inside each record's `process()` method — the audited individual `process()` impls in `records/ao.rs:344`, `records/bo.rs:235`, `records/calcout.rs:455`, and `records/waveform.rs` (no process method) compute the new value but do not touch `common.time`. Timestamp application is centralised in `src/server/database/processing.rs::process_record_with_links_inner`, where `apply_timestamp(&mut instance.common, is_soft)` is called at line 534, BEFORE the OUT-link write block (`src/server/database/processing.rs:575-651`), the `notify_from_snapshot` call (line 753), the OUT link write at line 758, and the FLNK propagation at line 807. Async writes also refresh the timestamp on completion (lines 1467, 1508). Downstream `TSEL` reads therefore see the new (current-cycle) timestamp; the C ordering bug is structurally avoided by centralised processing.
