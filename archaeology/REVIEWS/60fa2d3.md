The C bug was in `postfix.c` SEPERATOR (comma) and CLOSE_PAREN cases — both entered a loop that dereferenced `pstacktop->name[0]` without first checking that `pstacktop != stack` (the sentinel). On malformed input like `"0,"` or `"0)"`, the operator stack was at the base sentinel and the loop dereferenced uninitialized memory. In `epics-base-rs`, the equivalent module is `src/calc/engine/postfix.rs::compile`. Both cases use `stack.last()` returning `Option<&StackEntry>`, with explicit `None => return Err(...)` arms: `Token::RParen` returns `CalcError::ParenNotOpen` (line 373) and `Token::Comma` returns `CalcError::BadSeparator` (line 403). Vec underflow in the comma post-loop is also safe — after the loop `stack.last()` is the LParen so `stack.len() >= 1`. The fix is structurally already present via Rust's `Option`-returning `Vec::last`, and the error variants match what the C fix introduced.
