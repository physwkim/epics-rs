The C fix is in `dbAccess.c::dbPut`, which branches on `paddr->pfldDes->special == SPC_DBADDR` to enter a get_array_info / nRequest-clip / dbPutConvertRoutine / put_array_info pipeline keyed off a raw `void *paddr->pfield` pointer with an `offset` cursor.

`epics-base-rs` is a high-level Rust rewrite. The put path lives in `crates/epics-base-rs/src/server/database/field_io.rs` (`put_pv`, `put_pv_and_post_with_origin`, `put_record_field_from_ca`) and operates on `EpicsValue` enums dispatched through `Record::put_field(name, value)` — not `(void *paddr, offset, nRequest)` tuples. There is no `SPC_DBADDR` / `nRequest` / `offset` plumbing, no `dbPutConvertRoutine` table, no `get_array_info`/`put_array_info` callbacks, and no use-of-uninitialized-stack hazard (Rust would refuse to compile that). Long strings are owned `EpicsValue::String` / `CharArray` values; the array-element cursor model the C bug requires simply does not exist.

Field is `src/server/database/db_access.rs::db_put` per audit_targets, but no such function exists — `db_access.rs` is the in-process `DbChannel`/`DbSubscription` helper. The full record-write pipeline structurally cannot exhibit this bug.
